## Описание проекта

Проект реализует базовый пользовательский путь e-commerce:
1. просмотр каталога;
2. добавление товаров в корзину;
3. изменение количества;
4. расчет итоговой стоимости с учетом скидок и промокода.

Текущая реализация полностью клиентская: данные товаров берутся из `src/data/products.js`.

## Функции

### 1) Каталог товаров

**Кратко:** выводит карточки товаров с названием, изображением, ценой, скидкой и кнопкой добавления в корзину.

**Пример использования:**
1. Откройте страницу `/`.
2. Нажмите `Добавить в корзину` у нужного товара.
3. Если лимит товара достигнут, появится предупреждение.

### 2) Корзина

**Кратко:** показывает выбранные товары, позволяет менять количество, удалять позиции и видеть итоговую стоимость.

**Пример использования:**
1. Перейдите в `/cart`.
2. Измените количество товара через элементы управления в строке товара.
3. Проверьте пересчет суммы в блоке `Общая стоимость`.

### 3) Геокодинг адресов

**Кратко:** модуль для преобразования адреса пользователя в координаты для доставки (запланировано как внешняя интеграция).

**Пример использования (целевой сценарий):**
1. Пользователь вводит адрес на этапе оформления.
2. Приложение отправляет адрес в API геокодинга.
3. Полученные координаты используются для расчета зоны/стоимости доставки.

> Примечание: в текущем коде UI и вызовы геокодинга еще не реализованы.

### 4) Платежная система

**Кратко:** интеграция для создания платежа и подтверждения статуса оплаты (запланировано как внешняя интеграция).

**Пример использования (целевой сценарий):**
1. Пользователь нажимает `Оплатить заказ`.
2. Приложение запрашивает `paymentIntent`/ссылку на оплату через backend.
3. После подтверждения оплаты заказ переводится в статус `paid`.

> Примечание: в текущем коде checkout и платежный flow еще не реализованы.

## API интеграции

### Текущее состояние

- Приложение работает без backend API.
- Источник данных — локальный файл `src/data/products.js`.

### Целевая схема интеграций

#### Геокодинг API

- **Назначение:** валидация адреса и получение координат.
- **Типовой endpoint:** `POST /api/geocode`.
- **Вход:** строка адреса.
- **Выход:** нормализованный адрес, `lat`, `lng`, точность совпадения.

**Пример API-вызова:**

```bash
curl -X POST "https://api.example.com/api/geocode" \
  -H "Content-Type: application/json" \
  -d '{
    "address": "Москва, Красная площадь, 1"
  }'
```

**Пример ответа:**

```json
{
  "normalizedAddress": "Москва, Красная площадь, 1",
  "lat": 55.7539,
  "lng": 37.6208,
  "precision": "exact"
}
```

#### Payment API

- **Назначение:** создание платежа и проверка статуса.
- **Типовые endpoints:** `POST /api/payments/create`, `GET /api/payments/{id}`.
- **Вход:** сумма, валюта, состав заказа, идентификатор пользователя/гостя.
- **Выход:** идентификатор платежа, статус, ссылка или токен подтверждения.

**Пример API-вызова (создание платежа):**

```bash
curl -X POST "https://api.example.com/api/payments/create" \
  -H "Content-Type: application/json" \
  -d '{
    "amount": 89990,
    "currency": "RUB",
    "orderId": "order_12345"
  }'
```

**Пример ответа:**

```json
{
  "paymentId": "pay_9f8a7b6c",
  "status": "pending",
  "confirmationUrl": "https://pay.example.com/confirm/pay_9f8a7b6c"
}
```

```text
src/

    CartContext.jsx
  data/
```

#### Шаг 1. Клонирование репозитория

```bash

cd ai-project
```

#### Шаг 2. Установка зависимостей

```bash

```

#### Шаг 3. Настройка переменных окружения

Проект в текущем виде не требует обязательных секретов для локального запуска, но для API-интеграций обычно используются переменные окружения.

Пример установки переменных через shell:

```bash
export VITE_API_BASE_URL="https://api.example.com"
export VITE_GEOCODING_PROVIDER="provider-name"
export VITE_PAYMENT_PROVIDER="provider-name"
```

Пример использования в коде:

```js
const apiBaseUrl = process.env.VITE_API_BASE_URL;
```

```bash
npm run dev
```

После старта откройте адрес из терминала (обычно `http://localhost:5173`).

## Поддержка README в актуальном состоянии

Чтобы README всегда отражал текущее состояние проекта, используйте связку `git diff + AI + скрипты`.

### 1) Обязательное правило обновления

- Любое изменение в `src/contexts`, `src/components`, `src/pages`, `src/data` должно сопровождаться проверкой README.
- Если изменился пользовательский сценарий или API-контракт, обновляйте разделы `Функции`, `API интеграции`, `Примеры использования`, `FAQ`.

### 2) AI-процесс обновления документации

1. Получите список изменений: `git diff --name-only`.
2. Передайте diff в AI с запросом: "Обнови README по измененным файлам, не добавляя несуществующие функции".
3. Проверьте результат вручную по чеклисту:
   - все новые функции описаны;
   - удаленные функции убраны из README;
   - API-примеры соответствуют текущим контрактам;
   - раздел `Changelog` обновлен.

### 3) Рекомендуемые скрипты для автоматизации

Добавьте в `package.json` технические скрипты для CI/CD:

```json
{
  "scripts": {
    "docs:check": "node scripts/docs-check.mjs",
    "docs:sync": "node scripts/docs-sync.mjs"
  }
}
```

Минимальная логика:
- `docs:check`: валидирует, что ключевые разделы README существуют и не пустые.
- `docs:sync`: собирает изменения из git diff и формирует черновик обновления README.

### 4) Интеграция в Pull Request

- В шаблон PR добавьте пункт: `README updated?` (`yes/no + причина`).
- Для изменений API требуйте обновление раздела `API интеграции` как обязательное условие merge.
- В CI запускайте `npm run docs:check` вместе с тестами и линтерами.

## FAQ

### Где находится логика ограничений количества?

В `src/contexts/CartContext.jsx`.

### Почему в README есть геокодинг и платежи, если их нет в UI?

Это зафиксированная целевая архитектура интеграций для следующего этапа развития проекта.

### Где менять список товаров?

В `src/data/products.js`.

### Что делать, если `npm install` завершился с ошибкой?

Проверьте версию `Node.js` (`node -v`), удалите `node_modules` и `package-lock.json`, затем повторите `npm install`.

### Почему не открывается страница после `npm run dev`?

Убедитесь, что порт `5173` не занят, и откройте URL, который выводится в терминале.

### Где настроить URL API для интеграций?

В переменных окружения, например через `process.env.VITE_API_BASE_URL`.

### Геокодинг и платежи уже работают в приложении?

Пока нет: в текущей версии это описание целевой интеграции и готовый контракт для следующего этапа разработки.
